# 后续更新机制（公共）

业务模型保存到系统后，以下任一情况触发更新：

---

## 路径 A：GUI 页面修改自动触发

**触发条件**：对话框收到消息"业务模型文档已被更新"。

**🔴 执行规则（最高优先级）**：触发后**只输出**确认选项文字，**禁止输出任何其他内容**。

❌ **严格禁止在确认选项前/后输出以下任何内容**：
- 禁止解释"我看到您要求…"、"让我检查一下…"等说明性文字
- 禁止输出任何思考过程、推理、解释或进度说明

✅ **唯一允许的响应**：原样输出下方确认选项文字，一个字不改，不多加一个字。

### A1. 确认是否保存（必须执行，触发后立即输出，不得有任何前置或后置文字）

向用户展示选项（**以下文字必须原样输出，不可做任何修改**）：

```
文档已被更新，请选择：
1. 重新生成模型：依赖文档更新内容重新生成模型
2. 继续修改：返回调整分析结果

请问是否确认保存？
```

**根据用户回答执行**：
- 用户选择「保存」/「1」/「确认」/「是」→ 进入保存流程
- 用户选择「继续修改」/「2」/「还需要改」→ 等待用户输入，修改后再次询问

**用户确认后才执行保存流程**：**使用 Skill 工具调用** `save-model` skill 完成保存（🔴 必须通过 Skill 工具调用，不得在当前 session 内直接执行 save-model.md 的内容）。

**结果判断**：
save-model 执行完成后，读取 `output/save-model-result.json` 状态文件：
- `{"status": "error", ...}` → 向用户展示：`业务模型保存失败，请检查文档后重试`

---

## 路径 B：用户在交互中修改文档内容

**触发条件**：用户在对话中对业务对象或业务逻辑进行了修改并确认。

**执行流程**：
1. 根据修改的内容，对应变更 `output/stage2.md` 中的内容
2. **🔴 完整性检查**（必须执行，不可跳过）：
   - **属性完整性**：逐个检查所有已确认的业务对象，确保每个对象至少有一个属性。缺少则引导用户补充后再继续
   - **关系去重**：确认无互逆关系对、无完全重复的关系
3. 询问用户：文档内容已修改，是否同步变更到业务本体？
   - 回答是/确认，则执行下一步
   - 回答否，则不执行任何动作，结束流程
4. **使用 Skill 工具调用** `save-model` skill 完成参数构建和保存（🔴 必须通过 Skill 工具调用，不得在当前 session 内直接执行 save-model.md 的内容）
