
#### B1.存在基础模型

执行以下流程：

1. **生成业务模型**：获取基础模型 id，执行脚本 `python .claude/skills/ability-model/scripts/copyModel.py`
2. **写入状态文件**：根据脚本返回结果，静默写入 `output/save-model-result.json`：
   - 若 `success` 为 `True`：`{"status": "success", "modelPaperId": "<值>", "metaPaperId": "<值>"}`
   - 若 `success` 为 `False`：`{"status": "error", "message": "<错误信息>"}` → 终止流程，向用户提示：`业务模型初始化失败，请检查配置后重试`
3. **静默调用** 从输出中获取 `modelPaperId` 和 `metaPaperId`，**静默调用** MCP 工具 `mcp__oneui__sdk_tool`：
   * `module`: "action"
   * `api`: "trigger"
   * `payload`:` ["modelPaperAction", { "metaPaperId": "<脚本返回JSON中的metaPaperId>", "modelPaperId": "<脚本返回JSON中的modelPaperId>" }]`
     传参示例：
     ```json
     {
     "module": "action",
     "api": "trigger",
     "payload": [
     "modelPaperAction",
     {
     "metaPaperId": "56911",
     "modelPaperId": "56910"
     }
     ]
     }
     ```
⚠️ metaPaperId 和 modelPaperId 必填，值从脚本返回的 JSON 中提取。
❌ 禁止使用 module: "application" 或 api: "openApp"，这是错误的调用方式。
4. **用户提示**：`业务模型初始化已完成`，使用 Read 工具读取 `output/stage2.md`中的内容，展示生成的业务模型主要内容摘要，必须列出文档中所有对象

##### B2. 业务模型专业领域知识数据补充

1. **询问用户**
   ```
   业务模型初始化已完成。接下来需要基于已初始化的业务模型，补充具体的专业领域知识数据。

   专业领域知识数据能让业务模型更加完整和可落地，后续的交互逻辑也将依赖这些专业领域知识数据。
   ```

2. **提取业务模型实例**：

   **提取依据**（按优先级）：
   - 用户上传的原始文档中的具体案例
   - 使用 Read 工具读取 `output/stage1.md`,依据其中的业务场景描述，且对象范围严格限制在`output/stage2.md`中提到的业务模型中
   - 对话历史中提到的具体例子
   - 用户运行时上传的参考文档

      **语义映射规则（🔴 严格约束）**：
      - **在业务模型中已经明确定义里有哪些业务对象，每个业务对象又有明确的属性列表**
      - **必须提取`output/stage2.md`中定义的每一个业务对象的所有实例，不能遗漏任何对象**
      - **每个对象的每个属性都必须有对应的实例值，不能遗漏任何属性**
      - **映射时应理解字段的实际含义，而非机械匹配字面名称**
      - **（🔴 需要严格遵守）仅上下文中包含「需智」关键词的时，读取`.claude/skills/ability-model/references/instance-example.md`，遵守实例与模型对象的语义对应关系**

      **解析原则（🔴 严格约束）**：
      - **具象化原理**：模型对象可以被具象化为多个对象实例；例如："需求价值指标"对象 → 可具象化为"提升厂内清洁能源的使用占比"
      - **继承规范**：根据知识文档中的明确信息填充基础需智模型中的对象、属性、关系等数据结构;必须遵守模型中数据对象的属性和规则定义;**强制性的规则必须遵守**
      - **语义映射**：解析过程中，需要准确判断知识描述语意与模型对象之间的对应关系;可考虑构建常用语言映射表来辅助判断
      - **去重保关**：解析过程中，如果遇到重复的知识，则不需要多次填充，但需要保留该重复知识与其他对象之间的关系。示例： "空压机单耗指标"对象可能在"降本"和"节能"两个模糊意图中重复出现，只需保留一个对象，但关系中要明确关联到这两个模糊意图上。
      - **全量解析**：解析过程中，基础模型中的每一类对象都可能被具象化为多个对象实例，应该全部解析出来，不能只解析出一个就停止。示例： 一个知识文档中可能存在"降本"、"节能"、"低碳"等多个模糊意图；每个模糊意图下，又存在"空压机改造"、"节能服务"、"自建光伏"、"配置储能"等多领域的降本方式和对应的需求价值指标。
      - **适度推理**：文档解析过程中，允许基于知识原文做适当的总结、提炼和推理，但不要做过度分析延伸，原文没有提及的维度一定不要自主生成。
      - **多文档优先级**：当有多个不同类型的知识文档上传时，解析顺序为：总结型知识文档 > 案例型知识文档，后续解析作为之前解析的信息补充。
      - **案例知识提取**：如果解析的是案例型知识文档，需要注意填充模型时，不要直接填充案例的个性化数据信息，而是提取其中的知识内容。示例： 案例中提到该项目清洁能源利用率达到30%，则： - ✅ 清洁能源利用率 → 是知识（应提取） - ❌ 30% → 是项目个案信息（不应提取）

      **属性取值规则（🔴 严格约束）**：
      - **实例的属性取值必须符合 `output/stage2.md` 中各属性的"规则与约束"定义**
      - **实例必须符合 `output/stage2.md` 中的"业务对象实例生成规则与约束"**
      - **如果生成规则定义了特定的实例类型或范围，提取时必须遵循该约束**
      - **"特征标签"对象的标签取值可以留空（在模型调用时才会确定）**

      **二次校验规则（🔴 严格约束）**：
      - **关键信息完整性检查:确保知识文档中的所有关键信息都已经被提取并填充。**
      - **完备性检查:检查一个已填充的对象，其信息是否是符合一个完备知识点的要求。如果该知识点没有表达完善，可以利用大模型进行适当的推理补充。示例： 某个需求价值指标对象填充了指标项，但没有填充诊断规则、诊断参数、以及对应的特征标签等信息的情况。**
      - **语义偏差检查:检查一个已填充的对象，其填充的知识是否与对象定义存在语意理解偏差的情况。示例： 诊断指标、需求价值指标虽然都是指标形态，但其在需求解析的概念上存在重大差别，前者代表需求诊断参数或者客户特征标签，后者代表需求实现的量化标准。**
      - **关联关系检查:检查所有对象是否按照基础需智的模型规范进行填充，重点注意对象间关联关系是否存在遗漏。**
      - **个性化信息检查:确保所有对象中不含有知识文档中的案例个性化信息，例如案例的特征标签实际值、诊断参数实际值、需求价值指标实际值等。**


   **2.2提取示例**：

   - **例如已经有了一个【特征标签】的业务对象，以下在业务模型中该对象定义的属性列表。则最终提取的数据中，必须包含【特征标签】对象和【特征标签】的标签名称、标签维度、标签取值三个属性的实例数据**

     | 属性名称 | 属性定义                                                     | 规则与约束                                                             |
     | -------- | ------------------------------------------------------------ | ---------------------------------------------------------------------- |
     | 标签名称 | 明确标签名称，如“厂房屋顶载荷类型”                         | 同一目标客群的特征标签不可重复定义                                     |
     | 标签维度 | 为全方位、体系化构建特征画像而定义的特征标签通用维度分类方式 | 可以从行为偏好特征、商业运营特征、系统机理特征等维度选择，也可以自定义 |
     | 标签取值 | 明确标签的取值，如“屋顶面积=4000㎡”                        | 必须明确取值的单位                                                     |

   - **只提取 `output/stage2.md` 中明确提到的业务对象和业务逻辑的实例**
   - **不在 `output/stage2.md` 中的对象，即使文档中有相关数据，也不提取**
   - **严禁自行推测生成不存在的对象实例或属性取值**

   **2.3访谈策略**（只有当知识文档解析没有达到收敛条件时，才会触发引导问题）：
   - **从业务场景出发提问**
   - **避免直接问"有哪些实例"**
   - **问"在实际业务中，XX通常有哪些典型情况？能否举几个具体的例子？"**
   - **示例："在实际业务中，'模糊的意图'通常有哪些典型的情况？比如客户一般会提出什么样的需求？"**

   **置信度标注**：
   - **为每个实例标注来源和置信度**
   - **实例名称 #tag::success 高度确定#：来自用户明确提供的案例**
   - **实例名称 #tag::warning 需要确认#：从上下文推断，需用户确认**

3. - 请用户确认并接受反馈
   - 迭代修改直到用户明确表示不用修改

### 业务模型变更触发

**🔴 触发检查（每次接收到用户修改相关消息时执行）**：

当检测到用户消息中包含以下任一类操作时，触发后续更新流程：
- 新增、修改或删除了业务对象的属性（名称、描述、规则等）
- 新增、修改或删除了业务对象
- 新增、修改或删除了对象之间的关系（关联关系、映射关系等）
- 新增、修改或删除了业务逻辑规则
- 在对话中上传了业务模型文档

**执行流程（🔴 必须执行）**：
1. 根据 `update-mechanism.md` 的路径 B 执行
2. 询问用户：文档内容已修改，是否同步变更到业务本体？
3. 用户确认后调用 `save-model` skill

🛑 **STOP**：`B3` 更新完成后，返回主 SKILL.md 继续执行步骤 3 和步骤 4。
