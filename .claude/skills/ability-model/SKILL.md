# 业务模型生成子 Skill

你负责分析用户实现需求目标的过程涉及哪些业务对象和业务逻辑，以及它们之间的关系，生成业务模型。

## 核心目标
1. **业务对象**（🔴 所有领域都要分析）：业务实体，包含属性和规则
2. **业务逻辑**（🔴 所有领域都要分析）：描述业务对象之间动态关系的**原子性单一规则**，每条规则只做一件事——要么对属性进行计算，要么对条件进行判断
3. **对象关系**（🔴 所有领域都要分析）：
   - 固定关系：两个业务对象之间的静态关联
   - 动态关系：通过业务逻辑连接的关系
4. **对象实例**（⚠️ 可选，仅当用户明确提供时才写入文档）：
   - 业务对象的具体实例（真实业务数据案例，含属性取值）
   - 业务逻辑的具体实例（真实输入/输出案例）

### 业务逻辑的两种类型

- **计算类**：一个数学运算表达式（四则运算、求和、求均值等），计算的结果赋予对象属性
- **判断类**：一个条件判断（类似 SQL 的 WHERE 子句），满足条件时可执行对象属性的修改或对象间关系的操作

### 业务逻辑的输入输出

- **输入（input）**：数组，包含 1 个或多个业务对象（表示从该对象的属性中读取数据）
- **输出（output）**：数组，包含 1 个或多个业务对象或关系
   * 业务对象：表示对该对象的某个属性进行添加或修改
   * 关系：表示对两个对象之间的关系进行添加、修改或删除

❌ **业务逻辑不是 SOP 流程，不是多步骤操作序列，不是过程性编排描述**

## 注意事项

1. **你需要自己尝试分析**：而不是向用户直接询问答案
2. **只能询问帮助分析判断的信息**：如业务流程，而不是答案本身（如有哪些业务对象）
3. **每个对象必须和其他对象存在关系**：不允许出现无任何关联的对象
4. **如果存在基础模型**：最终的业务模型必须与基础模型完全一致，不允许删除、遗漏或添加业务模型中的任何一个对象或属性
5. **具体关联关系可以只写在一个对象下**
6. **🔴 对象分析的领域要求**：
    - **业务对象**：所有领域（基础领域和特定行业领域）都必须分析
    - **业务逻辑**：所有领域（基础领域和特定行业领域）都必须分析
    - 不管是什么领域，都要同时分析业务对象和业务逻辑
7. **业务逻辑的特殊要求**：
    - 必须是原子性规则：只允许计算类（单一数学表达式）或判断类（单一条件组合判断）
    - 必须明确标注 input（输入）和 output（输出）
    - ❌ 禁止将 SOP 流程、多步骤操作序列归入业务逻辑
8. **步骤标号展示规范**：向用户展示时，禁止使用任何步骤编码或步骤序号（包括 A1、A2、A3、A4、A5、B1、B2、B3、B4 及"步骤 1"、"步骤 2"等），只展示步骤内容本身
9. **🔴🔴🔴 属性完整性铁律**：所有业务对象必须至少包含一个属性，没有例外。在 A1 分析展示、A4 生成确认、A5.1 写入文档三个环节均须强制检查；发现无属性的对象，禁止继续后续步骤，必须先引导用户补充属性
10.**用户每次主动修改业务模型文档，必须询问用户：文档内容已修改，是否同步变更到业务本体？**

此 skill 必须严格遵守主 skill 中定义的全局铁律和通用规则：
