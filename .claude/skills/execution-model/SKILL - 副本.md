# 运行逻辑生成子 Skill

你负责生成智能工具的运行逻辑及规则，明确智能工具如何帮助用户实现需求目标，以及如何在过程中生成数据对象的具体实例。

## 核心目标

生成智能工具的运行逻辑及规则，包括：

1. **场景识别**：识别关键数据，并分析如何判定本体已沉淀完毕（内部理解为两个场景：沉淀本体 vs 运用本体）
2. **交互逻辑设计**：分析"沉淀本体"场景的具体业务流程和步骤（这是本次打造工具的核心交互逻辑）
3. **规则约束**：工具运行过程中必须遵守的全局规则

### 步骤 A2：识别关键数据

🔴 **核心目的**：识别业务模型中的关键数据，这是判定本体是否已沉淀完毕的依据

🔴 **内部理解**：智能工具的运行固定存在两个场景
- **场景 1（沉淀本体）**：通过交互帮助用户沉淀业务本体和经验
- **场景 2（运用本体）**：使用已沉淀的本体来处理具体业务

本步骤的任务是识别出"关键数据"，用于判定场景 1 是否已完成。

#### A2.1 分析并识别关键数据

从 stage2.md 的业务模型中识别出哪个对象是"关键数据"：

1. **关键数据的特征**：
   - 通常是业务模型中最核心的数据对象
   - 其他数据对象围绕它展开
   - 它的实例是否存在，标志着本体是否已沉淀完毕

2. **分析方法**：
   - 查看 stage2.md 中的所有数据对象
   - 分析对象之间的依赖关系
   - 识别出最核心、最关键的那个对象，往往处在图谱路径的“终点”

3. **向用户确认**：
   - 🔴 **使用业务语言**：不要说"核心对象"、"数据对象"等技术术语
   - 用"关键数据"、"核心信息"等业务人员能理解的表达
   - 示例："根据业务模型分析，'客户画像模板'是最关键的数据，其他信息都围绕它展开。您认为这个判断是否准确？"
   - 等待用户确认或修正

⚡ **停止点**：用户确认关键数据后，才能继续下一步

### 步骤 A3：确认场景识别规则

🔴 **核心目的**：基于关键数据，确认如何判定本体已沉淀完毕

#### A3.1 分析并提出识别规则

基于步骤 2 确认的关键数据，分析如何判定本体是否已沉淀：

1. **分析判定逻辑**：
   - 如果关键数据不存在实例 → 需要沉淀本体（场景 1）
   - 如果关键数据已存在实例 → 可以运用本体（场景 2）

2. **提出识别规则**：
   - 🔴 **必须提及 stage2.md 中存在的对象名**
   - 🔴 **使用业务语言表达**
   - 示例："当系统中还没有'客户画像模板'时，说明本体还未沉淀，需要通过交互来建立模板"

3. **向用户确认**：
   - 清晰说明识别规则
   - 等待用户确认或修正

⚡ **停止点**：用户确认识别规则后，才能继续下一步

### 步骤 A4：分析业务流程步骤

🔴 **核心目的**：分析"沉淀本体"场景的具体业务流程

#### A4.1 分析业务流程

分析该场景的业务流程：

1. **功能描述**：该场景需要实现的功能（即如何通过交互沉淀本体）

2. **流程步骤**：对每个步骤进行详细分析
   - **业务描述**：步骤的具体业务内容
   - **输入数据**：
     * 🔴 如果不为空，必须是《stage2.md》中定义的数据对象名称
     * 不应该提及上下文默认存在的内容（如对话内容、文档内容）
     * 只提及对象名称，不提及属性
   - **输出数据**：
     * 🔴 如果不为空，必须是《stage2.md》中定义的数据对象名称
     * 只提及对象名称，不提及属性
   - **注意事项**：步骤执行时需要注意的事项

3. **步骤顺序约束**：
   - 步骤的先后顺序要考虑数据对象之间的关系
   - 被依赖的对象应该先于依赖的对象被输出
   - 每个步骤应该和之前的流程有承接关系（步骤的输入应该是之前某些步骤的输出）

4. **步骤合并原则**：
   - 🔴 **避免步骤过于细碎**：如果多个步骤紧密相关、输入输出连续，应该合并为一个步骤
   - 🔴 **合并标准**：
     * 步骤之间没有明显的业务决策点
     * 步骤的输出直接作为下一步骤的输入，中间没有其他操作
     * 合并后的步骤仍然保持清晰的业务含义
   - 🔴 **目标**：整个流程的步骤数量应该控制在 5-8 个左右，最多不超过 10 个

#### A4.2 向用户展示业务流程

将业务流程清晰展示给用户：

1. **展示要求**：
   - 说明你的分析逻辑
   - 🔴 **使用业务语言**：每个步骤用连贯的业务语言描述
   - 🔴 **必须显示数据对象**：每个步骤要明确说明依赖哪些数据、产生哪些数据

2. **展示格式示例**：
   ```
   步骤 1：根据用户描述识别目标客群
   - 依赖数据：无
   - 产生数据：目标客群

   步骤 2：基于目标客群分析需求场景
   - 依赖数据：目标客群
   - 产生数据：需求场景

   步骤 3：结合需求场景提炼核心功能
   - 依赖数据：需求场景
   - 产生数据：核心功能
   ```

3. **禁止的格式**：
   ```
   ❌ 步骤 1：接收【用户描述】→ 输出【目标客群】
   ❌ 步骤 2：输入【目标客群】→ 输出【需求场景】
   ```

4. **用户确认**：
   - 请用户确认并接受反馈
   - 迭代修改直到用户明确表示不用修改

⚡ **停止点**：用户确认业务流程后，才能继续下一步
