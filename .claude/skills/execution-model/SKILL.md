# 交互逻辑生成子 Skill

你负责生成智能工具的交互逻辑及规则，明确智能工具如何帮助用户实现需求目标，以及如何在过程中生成数据对象的具体实例。

## 核心目标

生成智能工具的交互逻辑及规则，包括：

1. **交互逻辑设计**：分析具体的交互逻辑（这是本次打造工具的核心交互逻辑）
2. **规则约束**：工具运行过程中必须遵守的全局规则

### 步骤 A2：分析业务过程
🔴 **核心目的**：分析具体业务过程

#### A2.1 分析业务过程
从上下文中，分析业务过程、过程中包含业务环节和需要遵循的约束
1. **功能描述**：需要实现的功能

2. **业务环节**：对每个环节进行详细分析
   - **业务描述**：环节的具体业务内容
   - **数据依赖**：
     * 🔴 如果不为空，必须是《stage2.md》中定义的数据对象名称
     * 不应该提及上下文默认存在的内容（如对话内容、文档内容）
     * 只提及对象名称，不提及属性
   - **交付物**：
     * 🔴 如果不为空，必须是《stage2.md》中定义的数据对象名称
     * 只提及对象名称，不提及属性

3. **业务环节顺序约束**：
   - 业务环节的先后顺序要考虑数据对象之间的关系
   - 被依赖的对象应该先于依赖的对象被输出
   - 每个业务环节应该和之前的环节有承接关系（业务环节的数据依赖应该是之前某些业务环节的交付物）

4. **业务环节合并原则**：
   - 🔴 **避免业务环节过于细碎**：如果多个业务环节紧密相关、输入输出连续，应该合并为一个业务环节
   - 🔴 **合并标准**：
     * 业务环节之间没有明显的业务决策点
     * 业务环节的交付物直接作为下一业务环节的数据依赖，中间没有其他操作
     * 合并后的业务环节仍然保持清晰的业务含义
   - 🔴 **目标**：整个过程的业务环节数量应该控制在 5-8 个左右，最多不超过 10 个

#### A2.2 向用户展示业务过程

将业务过程清晰展示给用户：

1. **展示要求**：
   - 说明你的分析逻辑
   - 🔴 **使用业务语言**：每个业务环境用连贯的业务语言描述
   - 🔴 **必须显示数据对象**：每个业务环境要明确说明依赖哪些数据、产生哪些数据

2. **展示格式示例**：
   ```
   1：根据用户描述识别目标客群
   - 依赖数据：无
   - 产生数据：目标客群

   2：基于目标客群分析需求场景
   - 依赖数据：目标客群
   - 产生数据：需求场景

   3：结合需求场景提炼核心功能
   - 依赖数据：需求场景
   - 产生数据：核心功能
   ```

3. **禁止的格式**：
   ```
   ❌ 1：接收【用户描述】→ 输出【目标客群】
   ❌ 2：输入【目标客群】→ 输出【需求场景】
   ```

4. **用户确认**：
   - 请用户确认并接受反馈
   - 迭代修改直到用户明确表示不用修改

## 注意事项

1. **你需要自己尝试分析**：而不是向用户直接询问答案
2. **只能询问帮助分析判断的信息**：如业务过程，而不是答案本身（如所有的业务环节和规则约束）
3. **业务模型是最主要参考**：必须保证：
   - 🔴 **数据依赖、交付物约束**：业务环节的数据依赖和交付物如果不为空，就一定要和stage2.md中的数据对象名相符
   - 🔴 **数据对象覆盖约束**：stage2.md中的数据对象一定要被业务环节的输入、输出或者场景识别规则提及
   - 业务环节的先后顺序要考虑数据对象之间的关系（被依赖的对象应该先于依赖的对象被输出）
   - 每一个环节应该和之前的环节有承接关系（环节的数据依赖应该是之前某些业务环节的交付物）
   - 数据依赖、交付物中只提及对象，不提属性。而且不应该提及数据对象部分不存在的对象，比如对话内容、文档内容这些上下文默认存在，不需要作为输入或者输出列出来

4. **交互逻辑是通过用户与 Claude Code 之间的会话式交互来完成的**，所以必须分析出：
   - 引导对话的逻辑
   - 引导问题的生成逻辑
   - 数据对象实例的提取规则（如业务语言与数据对象的映射关系）

5. **每个环节都会输出一些数据对象**：需要分析每个环节的工作结果的展示规则

6. **🔴 分步交互**：
   - 必须按环节逐步完成，每个关键环节完成后必须停止
   - 停止点：识别关键数据后、确认识别规则后、确认业务过程后、确认规则约束后
   - 不能一次性生成所有内容让用户确认
   - 给用户展示执行过程时，不要直接展示内部环节标签，只显示环节后面的名称
     * ❌错误：`步骤 A2`：识别关键数据
     * ✅ 正确：识别关键数据

7. **🔴 使用业务语言**：
   - 向用户展示时，必须使用业务人员能理解的语言
   - 避免使用技术术语：
     * ❌ "核心对象"、"数据对象"
     * ✅ "关键数据"、"核心信息"、"沉淀本体"、"运用本体"
     * ❌ 禁止向用户展示 `stage1.md`、`stage2.md`、`stage3.md` 等技术文件名
     * ❌ 禁止向用户展示`constraintAppNo`、`分支A`、`分支B` 等技术词汇
   - 展示业务过程时，使用连贯的业务语言，并明确显示依赖和产生的数据
   - 示例：
     * ❌ 错误："步骤1：接收【用户描述】→ 输出【目标客群】"
     * ✅ 正确："1：根据用户描述识别目标客群\n- 依赖数据：无\n- 产生数据：目标客群"

8. **🔴 环节合并**：

   - 如果多个环节紧密相关、输入输出连续，应该合并为一个环节
   - 合并后的环节仍然保持清晰的业务含义
